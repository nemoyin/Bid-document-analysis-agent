## 二、发现的问题清单

### 🔴 严重 (Critical)

#### B1. total_duration_ms 计算错误（时区Bug）

| 项目 | 详情 |
|------|------|
| **位置** | `analysis_orchestrator.py:685-689` |
| **现象** | 存储的耗时 = 28,800,293ms（约8小时），实际耗时 = 293ms |
| **偏差** | 恰好 28,800,000ms = 8小时 = UTC+8时区偏差 |
| **根因** | `datetime.now(timezone.utc)` 创建的UTC时间与SQLite存储/读取之间的时区转换问题，timestamp() 计算结果被加了8小时偏移 |
| **影响** | 前端"总耗时"显示严重失真，用户以为分析跑了8小时 |

```python
# 问题代码
now = datetime.now(timezone.utc)
task.total_duration_ms = int(
    (now.timestamp() - task.started_at.timestamp()) * 1000
)
```

#### B2. 任务卡死无超时机制

| 项目 | 详情 |
|------|------|
| **位置** | `analysis_orchestrator.py:551-567` |
| **现象** | 任务 `ddc201c7...` 从 2026-06-25 12:47 开始一直处于 `analyzing` 状态，卡在 `image_similarity` 维度（running，0/3完成），已超过24小时 |
| **根因** | image_similarity 分析中出现未捕获异常或死循环，且无超时/心跳检测机制 |
| **影响** | 前端进度条永远停在45%，用户无法得知任务已死；数据库中存在"僵尸任务" |

#### B3. 维度内无细粒度进度更新

| 项目 | 详情 |
|------|------|
| **位置** | 各分析子模块 (text_similarity.py, image_similarity.py 等) |
| **现象** | 每个分析维度只有 `pending → running` 和 `running → completed` 两个状态跳变，`completed` 数始终为 0 直到维度完全完成 |
| **根因** | 子分析模块不回调进度更新接口，编排器仅在维度开始/结束时更新状态 |
| **影响** | 对于大文件（如100MB+的PDF），用户在维度分析期间看不到任何增量进度，体验极差 |

```python
# 目前：维度内 completed 始终为0
await self._update_dimension_status(task_id, "text_similarity", "running")
# ... 长时间分析，前端看到 completed=0/total=0 ...
await self._update_dimension_status(task_id, "text_similarity", "completed", completed=doc_pairs)
# 突然跳到 100%
```

### 🟠 重要 (Major)

#### B4. `issues_found` 字段永远为 0

| 项目 | 详情 |
|------|------|
| **位置** | `analysis_orchestrator.py:279-322` |
| **现象** | 分析全程 `issues_found` = 0，前端"已发现问题"卡片始终显示 0 |
| **根因** | `_count_issues_so_far()` 方法已实现但从未在任何分析阶段被调用 |
| **影响** | 用户无法实时了解分析过程中发现了多少问题 |

#### B5. `progress_detail.overall_progress` 永远为 0

| 项目 | 详情 |
|------|------|
| **位置** | `analysis_orchestrator.py:456-462` |
| **现象** | progress_detail JSON 中的 `overall_progress` 初始化后永不更新 |
| **根因** | 加权进度通过 `task.progress` 字段存储，但 `progress_detail` 内部字段未同步 |
| **影响** | 数据不一致，progress_detail 与 top-level progress 字段矛盾 |

#### B6. 进度展示"跳变"问题

| 项目 | 详情 |
|------|------|
| **位置** | `analysis_orchestrator.py:_calculate_weighted_progress` |
| **现象** | 进度从 0% 直接跳到 30%（文本完成），然后 45%（结构完成），再 60%... 呈阶梯状跳跃 |
| **根因** | 权重计算基于维度完成状态（0%或100%），无中间态 |
| **影响** | 进度条不是平滑增长，而是分段跳变，用户体验类似"卡顿" |

实际测试中的进度变化（高频率200ms轮询仍无法捕获中间态）：
```
[200ms]  progress=100%  ← 直接从 pending 跳到 completed（2个小文档 <300ms完成）
```

#### B7. `progress_only=true` 仍返回完整响应

| 项目 | 详情 |
|------|------|
| **位置** | `api/v1/analysis.py:176-189` |
| **现象** | `?progress_only=true` 返回了 22 个字段（含 `similarity_results`, `image_similarity_results` 等） |
| **根因** | `AnalysisTaskResponse.model_validate(task)` 可能因 `from_attributes=True` 拉取了ORM关联字段 |
| **影响** | 轮询数据量增大，大任务下网络开销不可忽略 |

### 🟡 一般 (Minor)

#### B8. 前端轮询无错误重试

| 项目 | 详情 |
|------|------|
| **位置** | `ProjectDetail/index.tsx:161-183` |
| **现象** | 轮询中任何网络错误直接 clearInterval 并停止轮询 |

| **影响** | 网络抖动可能导致进度展示永久中断，用户需要手动刷新 |

```typescript
} catch {
  if (pollRef.current) clearInterval(pollRef.current);
  setAnalyzing(false);  // 网络错误直接标记为分析结束
}
```

#### B9. 计时器精度问题

| 项目 | 详情 |
|------|------|
| **位置** | `AnalysisProgressPanel.tsx:35-46` |

| **影响** | 用户切到其他标签页再回来，看到的时间可能不准确 |

#### B11. 卡死任务的UI无反馈

| 项目 | 详情 |
|------|------|

## 三、代码质量评估

### 做得好的地方 ✅

1. **6维度加权进度计算** (`_calculate_weighted_progress`): 设计合理，权重可配置
2. **ETA 计算** (`_calculate_eta`): 基于剩余对比数×各维度耗时估算，逻辑清晰
3. **progress_only 轻量轮询**: API 设计了专门的轻量接口减少传输量（虽然当前实现有Bug）
4. **维度元数据集中管理**: `DIMENSIONS` 常量统一定义，前后端一致
5. **深拷贝保护**: `copy.deepcopy()` 确保 SQLAlchemy 检测到 JSON 字段变更
6. **完成后自动加载详情**: 前端在进度完成后自动调用 `getDetail` 获取完整结果

### 改进建议

| 优先级 | 建议 |
|--------|------|
| P0 | 修复 `total_duration_ms` 时区Bug（使用 `time.monotonic()` 或确保时区一致） |
| P0 | 添加分析任务超时机制（30分钟无进度更新→自动标记failed） |
| P1 | 在子分析模块中添加进度回调，实现维度内细粒度进度上报 |
| P1 | 在分析流程中定期调用 `_count_issues_so_far()` 更新 issues_found |
| P1 | 同步 `progress_detail.overall_progress` 与顶层 `progress` 字段 |
| P2 | 前端轮询增加错误重试逻辑（如连续3次失败才停止） |
| P2 | 修复 `progress_only=true` 返回冗余字段问题 |
| P2 | 前端增加"分析超时"警告提示（如5分钟无进度变化则警告） |
| P3 | 进度条增加 CSS transition 平滑过渡 |
| P3 | 计时器改用 `started_at` 差值计算（避免浏览器节流影响） |

## 四、测试结论

**大文件比对进度展示功能在基本流程上可工作，但存在多项影响用户体验的缺陷：**

1. **小文件场景**（<100KB, 2个文档）：分析在 300ms 内完成，进度条直接 0→100%，中间状态完全不可见。这是可接受的（太快了没必要展示进度）。

2. **大文件场景预期**：由于维度内无细粒度进度更新，用户将看到进度条长时间停滞在一个百分比，然后突然跳变。每个维度（特别是 error_consistency 预计最慢）会给人"卡住"的错觉。

3. **异常场景**：任务卡死无超时机制是最严重的问题。当前系统中已有一个"僵尸任务"卡了超过 24 小时。

4. **数据准确性**：`total_duration_ms` 存在 8 小时固定偏差的Bug，`issues_found` 始终为 0。

**综合评分**: 基础框架 ⭐⭐⭐⭐ | 细节完善 ⭐⭐ | 异常处理 ⭐
