# 招标文件合规性审查引擎 — 设计文档

**日期**: 2026-06-29
**版本**: v1.0
**项目**: BASS (Bid Analysis Supervision System)

---

## 1. 背景与目标

### 1.1 问题

现有 BASS 系统分析**投标文件**之间的相似度来检测围串标（投标人串通）。用户需要一个新的分析能力：审查**招标文件**本身是否合规——检测招标文件中是否存在违反招投标法的排他性条款、指向性技术要求、不合理资质门槛等"萝卜坑"行为。

### 1.2 目标

新增"招标文件合规审查"分析模式，上传一份招标文件 → 自动拆解条款 → 逐条判定合规风险（红/黄/绿）→ 输出综合合规评分 + 审查报告。

### 1.3 核心约束

- **规则动态可配置**：条款类型、违规规则、判定参数均可通过配置调整，无需改代码
- **风险评分+红黄灯**：与现有投标分析保持一致的结果形态
- **独立分析模式**：不与现有投标分析耦合
- **复用 Gitee AI API**：LLM 调用使用已有的 Gitee AI 服务

## 2. 整体架构

```
招标文件上传
    │
    ▼
┌──────────────────────────────────────┐
│  1. 文档解析层（复用现有 Parser）       │
│  PDF/DOCX → 全文文本 + 目录结构        │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  2. 条款结构化提取层（新增）             │
│  粗拆(规则) + 精拆(LLM) → 条款卡片列表  │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  3. 规则引擎层（新增，核心）             │
│  确定性规则(即时判定) + LLM语义规则      │
│  每条条款 → 🔴/🟡/🟢 + 违规依据         │
└──────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────┐
│  4. 综合评分 + 报告生成                 │
│  扣分模型 → 合规风险评分 → PDF/Word报告  │
└──────────────────────────────────────┘
```

**与现有系统的关系**：
- 复用 `BidDocument` 模型存储招标文件
- 复用 `document_parser` 解析 PDF/DOCX
- 复用 `AnalysisOrchestrator` 编排模式
- 复用 Gitee AI API 客户端
- **不修改**现有的投标分析逻辑

## 3. 条款结构化提取层

### 3.1 输入/输出

```
输入: 招标文件全文（50-200页）
输出: 条款卡片列表

条款卡片 = {
  id, 条款类型, 原文, 位置(章节/页码),
  关键参数: 按类型动态变化 (见下文)
}
```

### 3.2 条款类型（配置化）

存储在 `data/compliance_config.json`，可动态增删：

```json
{
  "clause_types": [
    {"id": "qualification", "label": "资格条件",   "active": true},
    {"id": "performance",   "label": "业绩要求",   "active": true},
    {"id": "technical",     "label": "技术要求",   "active": true},
    {"id": "timeline",      "label": "时间节点",   "active": true},
    {"id": "scoring",       "label": "评分标准",   "active": true},
    {"id": "contract",      "label": "合同条款",   "active": true},
    {"id": "other",         "label": "其他",       "active": true}
  ]
}
```

新增类型只需加一行，LLM prompt 动态拼接当前启用的类型列表。

### 3.3 两级提取策略

| 层级 | 方法 | 说明 |
|------|------|------|
| **粗拆** | 规则（目录+标题+编号模式） | 按章节/条款号切分为段落块，降低 LLM 处理量 |
| **精拆** | LLM 单次调用 + 结构化输出 | 从每个块识别条款类型、提取结构化参数 |

### 3.4 LLM 结构化输出 Schema

```python
{
    "clauses": [
        {
            "type": "qualification|performance|technical|...",
            "original_text": "原文段落",
            "location": {"chapter": "", "section": "", "page": 0},
            "params": {
                # 按类型动态变化
                "资质等级": "壹级",
                "项目数量": 3,
                "合同金额_万": 500,
                "品牌": "Intel",
                "型号": "Xeon Gold 6450",
                "天数": 3,
                "时间类型": "发售期"
            }
        }
    ]
}
```

## 4. 规则引擎层（核心）

### 4.1 规则模型

```
每条规则 =
  id, 名称, 类别, 默认风险等级(🔴/🟡/🟢), 权重, 启用状态
  适用条件: {条款类型, 触发关键词, 阈值参数}
  判定方式: deterministic | llm_semantic
  违规依据: [{法条名称, 法条内容}]
  prompt(仅LLM规则)
```

### 4.2 确定性规则（即时判定，零成本）

| ID | 规则 | 参数(可配) | 法条 |
|----|------|-----------|------|
| R01 | 发售期不足 | 最小天数=5 | 实施条例第16条 |
| R02 | 资质过高 | 资质等级阈值, 项目类型白名单 | 招投标法第18条 |
| R03 | 业绩门槛过高 | 金额倍数=2, 年限=3 | 实施条例第32条 |
| R04 | 品牌指向性 | 品牌数<3, 无"或同等"字样 | 实施条例第32条 |
| R05 | 主观分过高 | 主观分占比>40% | 招投标法第18条 |
| R06 | 资格预审排斥 | 预审条件与项目实际需要不符 | 招投标法第18条 |
| R07 | 保证金过高 | 保证金>2% | 实施条例第26条 |
| R08 | 特定地域限制 | 含"本地""本省""本地区"等 | 实施条例第32条 |

### 4.3 LLM 语义规则（处理需理解的场景）

| ID | 规则 | Prompt 要点 | 法条 |
|----|------|------------|------|
| R10 | 隐性排他条款 | 判断资质/技术/业绩条款是否隐性排除其他合格投标人 | 招投标法第18条 |
| R11 | 萝卜坑综合研判 | 综合多条条款，判断是否构成"量体裁衣"唯一指向 | 实施条例第32条 |
| R12 | 自定义规则 | 用户可新增 prompt 模板 + 指定条款类型范围 | 用户自定义 |

### 4.4 判定流程

```
每条条款卡片
    │
    ▼
遍历所有启用的规则
    │
    ├── 匹配适用条件? ──No──→ 跳过
    │
    ├── 确定性规则 ──→ 即时返回 🔴/🟡/🟢
    │
    └── LLM语义规则 ──→ 异步调用 Gitee AI → 🔴/🟡/🟢
    │
    ▼
汇总该条款所有命中规则 → 取最高风险等级
    │
    ▼
无命中 → 🟢 合规
```

### 4.5 规则可配置性

用户通过管理界面可以：
- 调整确定性规则的阈值参数
- 启用/禁用任意规则
- 调整每条规则的权重
- 新增 LLM 自定义规则（写 prompt + 引用法条）
- 克隆现有规则快速创建变体

## 5. 综合评分 + 报告

### 5.1 扣分模型

```
初始满分 100 分
├── 🔴 严重违规 (明确违反法条) → 每条 -15 ~ -25 分
├── 🟡 疑似违规 (需人工确认)   → 每条 -5 ~ -10 分
└── 🟢 合规                   → 不扣分

最终等级:
├── 85-100  → 🟢 低风险
├── 60-84   → 🟡 中风险
├── 40-59   → 🟠 高风险
├── 0-39    → 🔴 严重风险 (涉嫌量身定做)
```

### 5.2 报告结构

```
┌──────────────────────────────────┐
│  招标文件合规性审查报告            │
│  一、综合评分 + 风险等级           │
│  二、违规条款清单（按严重度排序）    │
│    每条款: 风险灯 + 违规依据 + 法条 │
│  三、条款全景 (全部条款红黄灯概览)   │
│  四、法条引用汇总                  │
└──────────────────────────────────┘
```

## 6. 数据模型

### 6.1 新表: `compliance_analyses`

| 列 | 类型 | 说明 |
|----|------|------|
| id | UUID PK | |
| project_id | FK→projects | 关联项目（独立分析模式，每个招标文件一个项目） |
| document_id | FK→bid_documents | 招标文件 |
| status | VARCHAR | pending/analyzing/completed/failed |
| compliance_score | DECIMAL(5,2) | 合规评分 0-100 |
| risk_level | VARCHAR | low/moderate/high/critical |
| clause_count | INT | 提取的条款总数 |
| violation_count | INT | 违规条款数 |
| details | JSON | 详细结果 |

### 6.2 新表: `compliance_clauses`

| 列 | 类型 | 说明 |
|----|------|------|
| id | UUID PK | |
| analysis_id | FK→compliance_analyses | |
| clause_type | VARCHAR | 条款类型 |
| original_text | TEXT | 原文 |
| location | JSON | 章节/页码 |
| params | JSON | 结构化参数 |
| risk_level | VARCHAR | 🔴/🟡/🟢 |
| matched_rules | JSON | 命中的规则列表 + 违规依据 |

### 6.3 新表: `compliance_rules`

| 列 | 类型 | 说明 |
|----|------|------|
| id | VARCHAR PK | 规则ID (如 "R01") |
| name | VARCHAR | 规则名称 |
| category | VARCHAR | deterministic/llm_semantic |
| is_active | BOOLEAN | 启用状态 |
| default_risk | VARCHAR | red/yellow/green |
| weight | DECIMAL(3,2) | 评分权重 |
| conditions | JSON | 适用条件 |
| params | JSON | 可配置参数阈值 |
| legal_basis | JSON | 引用法条 |
| llm_prompt | TEXT | LLM prompt 模板(仅 LLM 规则) |

## 8. 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| LLM | Gitee AI API (Chat Completion) | 已在项目中使用，复用 API Key |
| 条款提取 LLM | Qwen3 系列 (支持结构化输出) | 与 Embedding 使用同一平台 |
| 规则存储 | PostgreSQL JSON 列 | 灵活 schema，适合规则参数 |
| 配置管理 | `data/compliance_config.json` | 与现有 analysis_config.json 模式一致 |
| 前端框架 | React + Ant Design (复用) | 不引入新技术栈 |

## 9. 验证计划

1. 准备 3 份测试招标文件：合规文件、含排他条款的文件、含隐性指向文件
2. 单元测试：每个确定性规则的判定逻辑 + 边界值
3. 集成测试：完整流程（上传→提取→判定→评分→报告）
4. 人工验证：对比人工审查结果，计算准确率
5. 回归测试：确保现有投标分析不受影响
