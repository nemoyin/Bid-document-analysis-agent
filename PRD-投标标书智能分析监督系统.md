# 投标标书智能分析监督系统（MVP版）
## 产品需求文档 (PRD)

---

## 1. 项目信息

**项目名称**: 投标标书智能分析监督系统 (Bid Analysis Supervision System - MVP)

**项目代号**: BASS-MVP

**文档版本**: v1.1

**创建日期**: 2025-01-15

**产品负责人**: Alice (产品经理)

**技术栈**:
- 前端: React + Ant Design
- 后端: Python
- 文档解析: Apache Tika, POI, PDFBox, PaddleOCR
- 文本向量: Gitee AI Embedding API (Qwen3-Embedding-8B) + Cosine Similarity
- 向量数据库: ChromaDB（内置服务，MVP阶段<10万条向量）
- 图片分析: pHash, dHash, aHash

**原始需求概述**:
建设基于AI的投标文件分析平台，通过对多个投标人的标书进行自动解析和比对，发现围标、串标、陪标、关联投标等异常行为，形成监督线索。系统支持PDF、DOC、DOCX、ZIP格式的投标文件输入，输出相似度分析、异常特征分析、风险评分和围标预警。

**MVP范围边界**:
- ✅ **包含**: 标书上传、解析、多维度相似度分析、风险评分、报告生成
- ❌ **不包含**: 案件管理、工作流引擎、移动端支持、大模型问答功能

## 2. 产品定义

### 2.1 产品目标 (Product Goals)

1. **自动化检测异常投标行为**: 通过AI技术自动识别投标文件中的相似度异常、模板复用、错误一致性等围标串标特征，将人工审查时间减少80%

2. **提供量化风险评估**: 建立科学的风险评分模型（总分100分），将定性判断转化为定量指标，帮助监督人员快速定位高风险项目和企业

3. **生成可追溯的监督线索**: 自动生成结构化的相似性分析报告（PDF/Word/在线预览），为后续调查提供证据支持

### 2.2 用户故事 (User Stories)

**角色定义**:
- **主要用户**: 招投标监督人员、审计人员
- **次要用户**: 项目管理人员、法务人员

**核心场景**:

1. **As a 招投标监督人员**, I want 批量上传某个项目的所有投标文件并进行自动分析, so that 快速发现潜在的围标串标线索，无需逐份人工比对

2. **As a 招投标监督人员**, I want 查看企业之间的详细相似度对比（文本/图片/表格）, so that 准确定位异常相似的具体内容，形成可信的监督证据

3. **As a 审计人员**, I want 获得量化的风险评分和等级分类（低/中/高/严重风险）, so that 优先处理高风险项目，合理分配审计资源

4. **As a 项目管理人员**, I want 在首页查看项目整体风险态势（项目数量、标书数量、高风险统计）, so that 快速掌握监管工作的重点和难点

5. **As a 法务人员**, I want 导出正式的风险分析报告（PDF/Word格式）, so that 将技术分析结果转化为可用于法律程序的书面材料

## 3. 需求池 (Requirements Pool)

### P0 - 本期实现 (Must Have + 用户确认扩展功能)

#### 3.1 标书上传功能
- **REQ-001**: 系统必须支持单项目上传多个投标文件（PDF、DOC、DOCX格式）
- **REQ-002**: 系统必须支持批量上传（至少同时上传20个文件）
- **REQ-003**: 系统必须支持ZIP压缩包上传并自动解压
- **REQ-004**: 系统必须在上传后显示文件列表、上传进度、文件大小、格式验证结果
- **REQ-005**: 系统必须拒绝不支持的文件格式（除PDF/DOC/DOCX/ZIP外），并给出明确提示

#### 3.2 标书解析功能
- **REQ-006**: 系统必须能从PDF、DOC、DOCX文件中提取文本内容
- **REQ-007**: 系统必须能识别并提取标书的基础信息（投标人名称、项目名称、投标日期、金额等）
- **REQ-008**: 系统必须能区分并提取商务部分和技术部分的文本内容
- **REQ-009**: 系统必须能提取文件属性信息（作者、创建时间、修改时间、软件信息）
- **REQ-010**: 系统必须对扫描版PDF自动调用PaddleOCR进行文字识别
- **REQ-011**: 系统必须在解析失败时记录错误日志并通知用户

#### 3.3 文本相似度分析
- **REQ-012**: 系统必须计算任意两个标书之间的全文相似度（使用Gitee AI Embedding API（Qwen3-Embedding-8B）+ Cosine Similarity）
- **REQ-013**: 系统必须按章节（商务/技术）分别计算相似度
- **REQ-014**: 系统必须按段落计算相似度，并定位重复/高度相似的具体段落
- **REQ-015**: 相似度计算结果必须精确到小数点后2位（0-100%）

#### 3.4 风险评分模型
- **REQ-016**: 系统必须根据预设权重计算风险总分（0-100分）
  - 文本相似度: 30分
  - 目录结构相似: 15分
  - 图片相似: 15分
  - 表格相似: 10分
  - 错别字一致: 20分
  - 元数据一致: 10分
- **REQ-017**: 系统必须根据总分自动划分风险等级:
  - 0-30分: 低风险
  - 31-60分: 中风险
  - 61-80分: 高风险
  - 81-100分: 严重风险
- **REQ-018**: 风险评分必须在标书解析完成后5分钟内完成计算

#### 3.5 错误一致性分析（本期新增）
- **REQ-019**: 系统必须自动识别标书中的错别字、语病、漏字错误
- **REQ-020**: 系统必须比对不同标书中的错误是否一致（相同位置、相同错误类型）
- **REQ-021**: 错误一致性必须作为风险评分的重要依据（权重20分）

#### 3.6 图片相似分析（本期新增）
- **REQ-022**: 系统必须识别标书中的图片（项目组织架构图、流程图、示意图等）
- **REQ-023**: 系统必须计算图片相似度（使用pHash、dHash、aHash算法）
- **REQ-024**: 系统必须输出同源图片清单（相似度>90%标记为"同源"）
- **REQ-025**: 图片相似度必须纳入风险评分（权重15分）

#### 3.7 风险报告生成
- **REQ-026**: 系统必须能生成《投标文件相似性分析报告》PDF格式
- **REQ-027**: 系统必须能生成《投标文件相似性分析报告》Word格式
- **REQ-028**: 报告必须包含: 项目基本信息、投标企业列表、风险评分矩阵、相似度分析结果、异常特征清单
- **REQ-029**: 系统必须支持在线预览报告内容（无需下载）

## 6. 验收标准 (Acceptance Criteria)

### 6.1 功能验收

| 功能模块 | 验收标准 | 测试方法 |
|---------|---------|---------|
| 标书上传 | 支持PDF/DOC/DOCX/ZIP格式，单项目至少20个文件 | 边界值测试、格式验证测试 |
| 标书解析 | 文本提取准确率>95%（清晰文本），OCR识别准确率>85%（扫描版） | 抽样对比人工标注结果 |
| 文本相似度 | 相似度计算误差<±2%，高相似度（>80%）召回率>90% | 使用已知相似文档集测试 |
| 风险评分 | 评分计算正确，风险等级划分准确，评分时间<5分钟/项目 | 单元测试、性能测试 |
| 报告生成 | PDF/Word格式正确，内容完整，在线预览正常 | 格式验证、内容核对 |

### 6.2 性能验收

| 指标 | 要求 | 测试条件 |
|-----|------|---------|
| 文件上传速度 | <10秒/文件（50MB以内） | 100Mbps网络 |
| 标书解析速度 | <30秒/文件（100页以内） | 本地服务器部署 |
| 相似度分析速度 | <5分钟/项目（8个标书） | 并发数=1 |
| 系统并发能力 | 支持至少3个项目同时分析 | 服务器配置: 8核16G |

### 6.3 用户体验验收

- 所有页面响应式布局，支持1920×1080及以上分辨率
- 操作步骤不超过3步即可完成核心任务（上传→分析→查看报告）
- 错误提示信息清晰、可操作（不显示技术错误码）
- 报告生成后支持一键下载和分享

### 8.1 术语表

| 术语 | 定义 |
|-----|------|
| 围标 | 多个投标人事先串通，轮流中标或抬高标价的行为 |
| 串标 | 投标人与招标人串通，泄露标底或其他投标人信息的行为 |
| 陪标 | 投标人受雇于其他投标人，以陪衬身份参与投标的行为 |
| 关联投标 | 存在关联关系（如母子公司、同一实际控制人）的投标人同时参与同一项目投标 |
| Embedding | 将文本转换为向量表示的技朮，用于计算语义相似度 |
| Cosine Similarity | 余弦相似度，用于衡量两个向量方向上的相似性，取值范围[-1, 1] |
| pHash | 感知哈希算法，用于计算图片相似度，对缩放、压缩等变换具有鲁棒性 |
